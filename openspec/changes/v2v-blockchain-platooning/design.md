## Context

V2V（Vehicle-to-Vehicle）通信在自动驾驶编队场景下需要解决：
- **信任问题**：车辆间需要相互验证身份，防止恶意节点加入
- **数据完整性**：编队指令和状态同步需要防篡改
- **去中心化**：避免单点故障，编队Leader切换需要平滑
- **资源限制**：车载设备计算、存储、网络资源有限

传统区块链（如比特币、以太坊）过于重量级，不适用于车载边缘计算场景。需要设计一个轻量级、低延迟、适合移动节点的区块链系统。

## Goals / Non-Goals

**Goals:**
- 支持50-100个节点的车载区块链网络
- 区块确认延迟 < 100ms（编队控制实时性要求）
- 单节点存储 < 1GB（适配车载存储）
- 支持车辆动态加入/离开网络
- 实现Leader-Failover自动切换
- 提供完整的编队生命周期管理

**Non-Goals:**
- 不实现通用智能合约（仅支持预定义交易类型）
- 不与公链（以太坊等）互通
- 不处理车路协同（V2I）场景

## Decisions

### 1. 区块链类型：许可链（Permissioned Blockchain）
**决策**：使用许可链而非公链，所有节点需经CA认证后才能加入。

**理由**：
- 车载场景节点身份已知，无需匿名
- 许可链可以简化共识算法，提高效率
- 便于监管和审计

**替代方案**：公链（以太坊）- 因资源消耗和延迟过高而舍弃

### 2. 共识算法：PBFT
**决策**：采用PBFT（Practical Byzantine Fault Tolerance）共识算法，完整实现三阶段提交和视图变更。

**要求**：
- 最少4个节点（容忍1个故障）
- 推荐7个节点（容忍2个故障）
- 最大支持100个节点（其中7个Validator，其余为轻客户端Follower）

**已实现功能**：
- 完整的三阶段提交：PrePrepare → Prepare → Commit
- 视图变更（View Change）：Leader故障时自动切换
- 消息签名验证：防止恶意节点伪造消息
- Quorum机制：2f+1确认保证一致性

**约束**：
- 小编队（2-3辆车）不满足PBFT最低要求，需等待第4辆车加入后才能形成有效编队

**替代方案**（已舍弃）：
- Raft：因不支持拜占庭容错而舍弃
- PoA：因容错性不足而舍弃
- HotStuff：因复杂度较高而舍弃

### 3. 区块结构：轻量化设计
```
Block {
  Header {
    hash: 32 bytes          // 区块哈希
    prevHash: 32 bytes      // 前一区块哈希
    timestamp: 8 bytes      // Unix时间戳
    merkleRoot: 32 bytes    // 交易Merkle根
    validator: 20 bytes     // 出块节点地址
    signature: 65 bytes     // 区块签名
    txCount: 4 bytes        // 交易数量
  }
  Transactions []          // 交易列表（最大100条/块）
}
```

**理由**：最小化存储，每个区块<10KB，适配车载设备。

### 4. 网络层：Libp2p + Gossipsub
**决策**：使用Libp2p作为P2P网络基础，Gossipsub进行消息传播。

**理由**：
- Libp2p支持NAT穿透，适应移动网络
- Gossipsub提供高效的广播机制
- 支持Topic订阅，便于分区通信

### 5. 存储：LevelDB + 轻客户端模式
**决策**：使用LevelDB存储区块链数据，支持轻客户端同步（SPV模式）。

**理由**：
- LevelDB轻量、嵌入式、适合边缘设备
- 轻客户端只存储区块头，跟随者节点可节省90%存储

### 6. 编队拓扑：Leader-Validator-Follower
**决策**：编队由Leader、Validators和Followers组成。

**角色定义**：
- **Leader（1个）**：负责出块，是PBFT的Primary节点
- **Validators（6个）**：PBFT共识参与者（Replicas），验证和确认区块
- **Followers（0-93个）**：轻客户端，只同步区块头，不参与共识

**编队规模**：
- 最少4辆车（1 Leader + 3 Validators，此时容错f=1）
- 推荐7辆车（1 Leader + 6 Validators，此时容错f=2）
- 最大100辆车（7 Validators + 93 Followers）
- **约束**：少于4辆车无法形成有效编队

**Validator选择策略**：
- 编队创建时，从所有车辆中随机选择7个作为Validators
- 编队规模小于7时，所有车辆都作为Validators
- Validator故障时，从Followers中随机选择补充

**Leader故障处理**：
- 通过PBFT的View Change机制选举新Leader
- 视图变更时间<10秒

| 用途 | 语言 | 版本 | 说明 |

- 区块链生态成熟（geth、cosmos-sdk等参考实现）

### 7. 关键技术选型理由

#### 7.1 为什么选 Go 而不是 Rust/C++?

| 对比项 | Go | Rust | C++ |
|--------|-----|------|-----|
| 开发效率 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| 性能 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 内存安全 | ⭐⭐⭐⭐ (GC) | ⭐⭐⭐⭐⭐ (所有权) | ⭐⭐⭐ (手动) |
| 区块链生态 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 车载部署 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 团队学习成本 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |

**结论**: Go 在开发效率、生态、学习成本上最优，性能满足需求(<100ms)。

#### 7.2 为什么选 LevelDB 而不是 SQLite/Badger?

| 特性 | LevelDB | SQLite | Badger |
|------|---------|--------|--------|
| 类型 | LSM-Tree KV | SQL DB | LSM-Tree KV |
| 写性能 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 读性能 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 资源占用 | ⭐⭐⭐⭐⭐ (低) | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 嵌入式 | ✅ | ✅ | ✅ |
| 区块链生态 | ⭐⭐⭐⭐⭐ (geth/eth) | ⭐⭐⭐ | ⭐⭐⭐ |

**结论**: LevelDB 轻量、写性能优，是区块链主流选择。

#### 7.3 为什么选 libp2p 而不是自研P2P?

- 支持多种传输协议(TCP/QUIC/WebSocket)
- 内置NAT穿透、节点发现
- 成熟的Gossipsub广播机制
- 区块链项目广泛采用(Filecoin, Polkadot等)
- 节省开发成本(预估6个月自研 vs 2周集成)

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| [风险] 车辆高速移动导致网络分区 | 实现分区恢复机制，使用CRDT合并冲突状态 |
| [风险] 共识延迟影响编队控制实时性 | 关键控制消息走快速通道（先执行后共识），非关键消息异步确认；限制编队最小规模为4辆车 |
| [风险] 车载设备断电/重启导致数据丢失 | 定期Checkpoint + 状态快速同步 |
| [风险] 证书泄露导致身份伪造 | 实现证书吊销列表（CRL）和短有效期证书自动轮换 |
| [风险] 存储持续增长 | 定期归档历史数据到云端，本地只保留最近N个区块 |

## Migration Plan

1. **阶段1**：单机测试网络部署，验证核心功能
2. **阶段2**：双车场地测试，验证V2V通信和共识
3. **阶段3**：多车编队测试（5-10辆），验证规模和性能
4. **阶段4**：实际道路测试，验证可靠性和安全性

**Rollback策略**：保留传统编队控制作为fallback，区块链异常时自动降级。
