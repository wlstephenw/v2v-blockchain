# V2V Blockchain 架构设计文档

## Task 13.6: 架构设计文档

## 1. 概述

V2V Blockchain 是一个专为车联网 (V2V) 场景设计的轻量级区块链系统，支持车辆编队管理和安全通信。

### 1.1 设计目标

- **高吞吐**: 支持 1000+ TPS，满足车辆高频通信需求
- **低延迟**: 区块确认时间 < 2秒，确保实时性
- **轻量级**: 支持资源受限的车载设备
- **安全性**: 基于身份的加密和 PBFT 共识
- **可扩展**: 支持动态节点加入和编队管理

### 1.2 核心特性

- 轻量级区块链核心
- P2P 网络通信
- PBFT 共识机制
- 车辆身份认证
- 编队管理
- 消息验证和加密
- 状态追溯和审计

## 3. 区块链核心

### 3.1 区块结构

```go
type Block struct {
    Header       *BlockHeader
    Transactions []*Transaction
    Signature    []byte
}

type BlockHeader struct {
    Height       uint64
    PrevHash     Hash
    Hash         Hash
    Timestamp    int64
    MerkleRoot   Hash
    Validator    Address
    Signature    []byte
}
```

### 3.2 交易类型

| 类型 | 说明 | 用途 |
|------|------|------|
| Transfer | 转账交易 | 车辆间价值转移 |
| Register | 注册交易 | 车辆身份注册 |
| Platoon | 编队交易 | 编队创建/加入/离开 |
| Data | 数据交易 | 传感器数据上链 |

### 3.3 共识机制 (PBFT)

```
┌─────────┐     Pre-prepare      ┌─────────┐
│ Primary │ ───────────────────> │ Replica │
│  (Leader)│                     │         │
└─────────┘                     └─────────┘
       │                              │
       │        Prepare               │
       │ <──────────────────────────> │
       │                              │
       │         Commit               │
       │ <──────────────────────────> │
       │                              │
       ▼                              ▼
  ┌─────────┐                   ┌─────────┐
  │Committed│                   │Committed│
  └─────────┘                   └─────────┘
```

## 5. 编队管理

### 5.1 编队生命周期

```
    ┌─────────┐
    │  Forming │
    │ (创建中) │
    └────┬────┘
         │ 成员 >= 4
         ▼
    ┌─────────┐
    │  Active │
    │ (运行中) │
    └────┬────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
┌────────┐ ┌─────────┐
│ Dissolved│ │ Leader  │
│ (已解散) │ │ Change  │
└────────┘ └────┬────┘
                │
                ▼
           ┌─────────┐
           │  Active │
           │ (新Leader)│
           └─────────┘
```

### 5.2 角色定义

| 角色 | 职责 | 数量 |
|------|------|------|
| Leader | 出块、协调编队 | 1 |
| Validator | 参与共识、验证区块 | 3+ |
| Follower | 同步区块、不参与共识 | 0+ |

### 5.3 Leader 选举

```
1. 当前 Leader 故障检测 (连续3周期未出块)
2. 触发 PBFT View Change
3. 新 View 编号 = 旧 View + 1
4. 新 Leader = Validators[ViewNumber % ValidatorCount]
5. 新 Leader 开始出块
```

## 6. 安全机制

### 6.1 身份认证

```
┌──────────┐      ┌──────────┐      ┌──────────┐
│   CA     │─────►│ Certificate│─────►│  Vehicle │
│ (Root)   │      │  (Signed)  │      │  (Store) │
└──────────┘      └──────────┘      └──────────┘
       │                                  │
       │         Verify Chain             │
       └──────────────────────────────────┘
```

### 6.2 消息签名

- 算法: ECDSA (secp256k1)
- 哈希: Keccak-256
- 签名长度: 65 bytes

### 6.3 加密通信

- 端到端加密: ECIES
- 群组加密: AES-256-GCM + 群组密钥
- 密钥交换: ECDH

| 数据类型 | 保留时间 | 处理方式 |
|---------|---------|---------|
| 完整区块 | 7天 | 本地存储 |
| 区块头 | 1年 | 本地存储 |
| 审计日志 | 1年 | 本地 + 归档 |
| 事件数据 | 30天 | 自动清理 |

## 8. 性能指标

### 8.1 目标性能

| 指标 | 目标值 | 说明 |
|------|--------|------|
| TPS | 1000+ | 每秒交易数 |
| 区块时间 | 2s | 平均出块间隔 |
| 确认时间 | < 4s | 交易最终确认 |
| 节点同步 | < 30s | 新节点同步时间 |
| 存储增长 | < 1GB/天 | 日数据增长 |

| 节点类型 | CPU | 内存 | 存储 | 带宽 |
|---------|-----|------|------|------|
| Validator | 2核 | 2GB | 10GB+ | 100Mbps |
| Follower | 1核 | 1GB | 5GB+ | 50Mbps |

## 11. 故障处理

### 11.1 故障检测

| 故障类型 | 检测方式 | 处理策略 |
|---------|---------|---------|
| Leader 故障 | 超时检测 | View Change |
| 网络分区 | 心跳检测 | 分区恢复 |
| 拜占庭节点 | 行为分析 | 隔离告警 |
| 存储故障 | 写入检测 | 切换备用 |

### 11.2 恢复机制

- 自动重连
- 状态同步
- 数据修复
